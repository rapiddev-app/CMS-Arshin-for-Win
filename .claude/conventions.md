# Правила разработки кода

> Этот документ содержит ключевые правила для генерации кода. Полное техническое видение проекта см. в [@vision.md](vision.md)

## Принципы

### KISS (Keep It Simple, Stupid)
- Простые решения вместо сложных
- Если можно решить в 10 строк - не пишем 100
- Никаких абстракций "на вырост"
- Не добавляем функциональность "на всякий случай"

### Никакого оверинжиниринга
- Не используем паттерны проектирования там, где они не нужны
- Решаем конкретную задачу конкретным способом
- Три похожие строки кода лучше, чем преждевременная абстракция

### Читаемость кода
- Понятные названия переменных и функций на русском языке
- Один модуль = одна зона ответственности
- Короткие функции с понятной логикой
- Комментарии на русском языке только где логика неочевидна

### Минимум зависимостей
- Используем встроенные библиотеки Python где возможно
- Сторонние библиотеки только когда действительно необходимо

## Обработка ошибок

### Fail-fast принцип
- Если что-то не так - сразу показываем ошибку
- Простые сообщения об ошибках на русском языке
- Валидация входных данных перед выполнением

### Telegram уведомления
Все ошибки отправляются в Telegram:
```python
try:
    # код
except Exception as e:
    self.send_telegram_notification(f"ОШИБКА: {e}", to_group=True)
    raise
```

### Защита от дублей
Обязательная проверка существования файла:
```python
if os.path.exists(filepath):
    error_msg = f"Дубль файла {filename}, проверь! Реестр не отправлен!"
    self.send_telegram_notification(error_msg, to_group=True)
    raise FileExistsError(error_msg)
```

## Конфигурация

### JSON как источник правды
- Все настройки в JSON файлах (см. [@vision.md](vision.md#модель-данных))
- Только `config_manager.py` имеет право читать/писать конфиги
- Скрипты читают конфиги в read-only режиме

### Пути к файлам
- Используем абсолютные пути из конфигов
- Не хардкодим пути в коде

## Планирование задач

### Умное расписание
После успешной отправки скрипт ОБЯЗАТЕЛЬНО пересчитывает следующий запуск:
```python
def run(self):
    try:
        data = self.gather_data()
        if data is not None and len(data) > 0:
            filename, filepath = self.create_output_file(data)
            self.send_email(filename, filepath)
            self.send_telegram_notification(f"Файл {filename} успешно отправлен", to_group=True)

            # ВАЖНО: Обязательный пересчет следующего запуска
            self.reschedule_next_run()
```

### Проверка календаря
См. алгоритм в [@vision.md](vision.md#проверка-производственного-календаря):
- Проверка выходных (суббота, воскресенье)
- Проверка праздников из `calendar.json`
- Поиск следующего рабочего дня


## Работа с датами

### Форматирование
- Входные данные: `pd.to_datetime(df['Дата'], errors='coerce')`
- Вывод в Excel: `df['Дата'].dt.strftime('%d.%m.%Y')`
- Вывод в имя файла: `date.strftime('%d.%m.%y')`

### Валидация дат
Обязательная проверка диапазона:
```python
today = datetime.now().date()
min_valid_date = today - timedelta(days=30)

if (date > today) or (date < min_valid_date):
    send_telegram_message("Дата вне допустимого диапазона", to_group=True)
    return pd.DataFrame(), []
```

## Что НЕ делать

❌ Не использовать паттерны проектирования без необходимости
❌ Не создавать абстракции "на вырост"
❌ Не добавлять функциональность "на всякий случай"
❌ Не использовать async/await (не требуется для задачи)
❌ Не использовать базы данных (только JSON)
❌ Не использовать LLM (см. [@vision.md](vision.md#работа-с-llm))
❌ Не добавлять docstrings к коду, который не изменялся
❌ Не рефакторить код, который работает
❌ Не использовать английские комментарии (только русские)
