# Проектные правила ScriptManager

> **Обязательные к исполнению правила для всех участников разработки**

## 📋 Документация проекта

Перед началом работы изучить:
1. **[@.claude/workflow.md](.claude/workflow.md)** - процесс разработки (КАК работаем)
2. **[@.claude/conventions.md](.claude/conventions.md)** - правила кода (ЧТО пишем)
3. **[@doc/vision.md](doc/vision.md)** - техническое видение (ЗАЧЕМ и КУДА)
4. **[@doc/tasklist.md](doc/tasklist.md)** - план работ (ЧТО делаем)

## ⚡ Главные правила

### 1. Всегда следовать workflow
**Обязательно:** [@.claude/workflow.md](.claude/workflow.md)

- ❌ **НЕ писать код** без согласования решения
- ❌ **НЕ переходить** к следующей итерации без подтверждения
- ✅ **Согласовать → Реализовать → Протестировать → Подтвердить**

### 2. Писать код по conventions
**Обязательно:** [@.claude/conventions.md](.claude/conventions.md)

- KISS принцип - простые решения
- Комментарии на русском
- Обработка ошибок с Telegram уведомлениями
- Логирование ключевых операций

### 3. Работать по плану
**Обязательно:** [@doc/tasklist.md](doc/tasklist.md)

- Выполнять итерации последовательно (0 → 1 → 2 → ... → 8)
- Не пропускать итерации
- Обновлять прогресс после каждой итерации

---

## 🔄 Процесс работы (краткая справка)

### Перед началом итерации:
1. Прочитать описание итерации из [@doc/tasklist.md](doc/tasklist.md)
2. Изучить требования из [@doc/vision.md](doc/vision.md)
3. Предложить решение (структура + код)
4. **ЖДАТЬ ПОДТВЕРЖДЕНИЯ**

### После подтверждения:
1. Написать код по [@.claude/conventions.md](.claude/conventions.md)
2. Написать тест
3. Запустить тест
4. Показать результаты
5. **ЖДАТЬ ПОДТВЕРЖДЕНИЯ**

### После успешного теста:
1. Обновить прогресс в [@doc/tasklist.md](doc/tasklist.md)
2. Отметить задачи чекбоксами
3. Спросить разрешение на переход к следующей итерации
4. **ЖДАТЬ ПОДТВЕРЖДЕНИЯ**

## 🚫 Запрещено

### Категорически запрещено:
- Писать код без предварительного согласования
- Переходить к следующей итерации без подтверждения
- Добавлять функциональность вне плана
- Рефакторить работающий код без причины
- Использовать английские комментарии
- Пропускать обновление прогресса
- Использовать паттерны проектирования без необходимости

### Требуется согласование:
- Любые отклонения от согласованного решения
- Добавление новых зависимостей
- Изменение архитектуры
- Изменение структуры данных

---

## ✅ Обязательно

### В каждом файле кода:
- Комментарии на русском языке
- Обработка ошибок через try/except
- Логирование важных операций
- Docstrings для функций и классов

### В каждой итерации:
- Тест работоспособности
- Обновление [@doc/tasklist.md](doc/tasklist.md)
- Соответствие [@.claude/conventions.md](.claude/conventions.md)
- Подтверждение на переход

## 📝 Шаблоны

### Предложение решения:
```
Итерация X: [Название]

Планирую создать:
- файл 1 - назначение
- файл 2 - назначение

Ключевые компоненты:
- class/def описание

Подход:
- описание логики

Подходит?
```

### Отчет о тесте:
```
Тест итерации X:

Команда: [команда]
Ожидаемый результат: [описание]
Фактический результат: [описание]

Статус: ✅ Успешно / ❌ Ошибка
```

### Переход к следующей итерации:
```
Итерация X завершена и протестирована.
Прогресс: X/8 (XX%)

Готов начать Итерацию X+1: [Название]?
```

- [ ] Код соответствует [@.claude/conventions.md](.claude/conventions.md)

- [ ] Прогресс обновлен в [@doc/tasklist.md](doc/tasklist.md)

## 💡 Девиз проекта

**"Сначала согласуем, потом реализуем, затем тестируем, после подтверждаем"**

**KISS принцип:** Простые решения, минимум абстракций, работает > идеально

---

## ⚠️ Важно помнить

1. **Workflow** - это не рекомендация, это обязательный процесс
2. **Conventions** - это не пожелания, это строгие правила
3. **Tasklist** - это не примерный план, это точный маршрут
4. **Подтверждения** - это не формальность, это контрольные точки

Любое отклонение от этих правил должно быть согласовано явно.
